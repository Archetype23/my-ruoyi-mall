## 1. 改造概述

### 1.1 原系统架构

| 组件 | 原系统 |
|------|--------|
| 后端框架 | ruoyi-vue-pro (Spring Boot 3 + MyBatis Plus) |
| 管理后台前端 | yudao-ui-admin-vue3 (Vue 3 + Element Plus) |
| 用户端前端 | yudao-ui-user-vue3 (Vue 3) |
| 数据库 | MySQL 8.0 |
| 缓存 | Redis |

### 1.2 改造目标

在原有商城系统基础上，新增**社区团购**功能模块，支持：
- 团购活动发布与管理
- 用户参团/开团
- 自提点管理
- 团购订单处理

### 1.3 改造范围

| 模块 | 改造内容 |
|------|----------|
| 数据库 | 新增 4 张团购业务表，修复多张表结构 |
| 后端 | 新增团购 Activity/Record 模块，修复多租户兼容 |
| 前端（管理端） | 新增团购管理页面，精简菜单结构 |
| 前端（用户端） | 新增团购浏览/参团页面 |

## 5. 遇到的问题与解决方案

### 5.1 构建与环境问题

#### 问题 1：FUSE 文件系统导致 `mvn clean` 失败

**现象**：
```
rm: cannot remove '.fuse_hidden...': Device or resource busy
```

**原因**：Podman 挂载的 FUSE overlay 文件系统与 `mvn clean` 不兼容。

**解决**：改用 `mvn install` 跳过 clean 阶段。

---

#### 问题 2：前端容器 CORS 错误

**现象**：浏览器请求 `host.containers.internal:48080` 被拒绝。

**原因**：浏览器无法解析容器内部 hostname。

**解决**：删除 `-e VITE_BASE_URL` 覆盖，使用 `.env.local` 中的 `localhost:48080`。

---

### 5.2 多租户问题（核心问题）

#### 问题 3：18 张表缺少 `tenant_id` 列

**现象**：
```
Unknown column 'tenant_id' in 'where clause'
```

**原因**：ruoyi-vue-pro 多租户框架默认对所有表注入 `WHERE tenant_id = ?`，但建表 SQL 未包含该列。

**解决**：为 18 张 pay/trade/promotion 表添加 `tenant_id` 列。

---

#### 问题 4：缺失整张表

**现象**：
```
Table 'ruoyi-vue-pro.promotion_combination_record' doesn't exist
```

**原因**：原系统 SQL 文件未包含这些表的建表语句。

**解决**：创建 7 张缺失表：
- `promotion_combination_record`
- `trade_statistics`
- `product_statistics`
- `member_config`
- `member_point_record`
- `pay_wallet_recharge`
- `trade_delivery_pick_up_store`

---

#### 问题 5：缺失字段

**现象**：
```
Unknown column 'transfer_notify_url' in 'field list'
Unknown column 'delivery_types' in 'field list'
Field 'bar_code' doesn't have a default value
```

**原因**：Java 实体类有字段，但数据库表缺少对应列或列约束过严。

**解决**：
- 添加 `pay_app.transfer_notify_url`
- 添加 `product_spu.delivery_types`
- 修改 `product_spu.bar_code` 为可空
- 修改 `product_spu.unit` 等字段添加默认值
- 添加 `product_sku.first_brokerage_price`、`second_brokerage_price`
- 修改 `member_level.icon`、`background_url` 为可空

---

### 5.3 代码问题

#### 问题 6：团购实体类继承错误

**现象**：团购模块数据无法按租户隔离。

**原因**：`GroupBuyActivityDO` 和 `GroupBuyRecordDO` 继承 `BaseDO` 而非 `TenantBaseDO`。

**解决**：修改继承关系为 `extends TenantBaseDO`。

---

#### 问题 7：统计页面 NPE

**现象**：
```
NullPointerException: Cannot invoke "java.time.LocalDateTime.minusDays(long)" because "<parameter1>" is null
```

**原因**：前端调用 `/statistics/member/analyse` 和 `/statistics/product/analyse` 时未传 `times` 参数。

**解决**：在 Controller 中添加 null 检查，设置默认时间范围（近 30 天）。

---

### 5.4 菜单与缓存问题

#### 问题 8：菜单过多且包含无用模块

**现象**：管理后台显示 BPM、CRM、ERP、AI、IoT 等 400+ 条菜单。

**解决**：
- 删除约 100 条无用菜单
- 保留团购相关的 4 大模块
- 执行 `FLUSHALL` 清除 Redis 缓存
- 重启后端重新加载菜单

---

### 5.5 前端连接问题

#### 问题 9：用户端 API 前缀错误

**现象**：用户端请求 `/admin-api/...` 而非 `/app-api/...`。

**解决**：修改 `.env.local` 中 `VITE_API_URL=/app-api`。

---

#### 问题 10：用户端登录字段不匹配

**现象**：登录时返回错误。

**原因**：前端发送 `username`，后端期望 `mobile`。

**解决**：修改 `login/index.ts` 字段映射。

## 7. 角色分离实现

### 7.1 概述

管理后台实现了管理员/商家角色分离：
- **管理员（admin）**：拥有全部 5 个顶级菜单（系统管理、基础设施、支付管理、会员中心、商城系统）
- **商家（merchant）**：仅拥有 3 个业务菜单（支付管理、会员中心、商城系统）

### 7.2 实现方式

1. 商家角色（ID=161, code=merchant）通过 `system_role_menu` 表分配菜单权限
2. 采用递归 CTE 方式，将 支付管理(1117)、会员中心(2262)、商城系统(2362) 的所有子菜单分配给商家角色
3. 后端 `@PreAuthorize` 权限注解自动拦截商家对系统管理/基础设施的访问（返回 403）
4. 前端菜单树由 `/system/auth/get-permission-info` 接口根据角色过滤后返回
