## 3. 正向闭环

1. 管理后台创建或确认一个进行中的社区拼团活动。
   - 拼团价、原价、成团人数、库存、成团时限正确。
   - 活动状态为“进行中”，库存未售罄。

2. 用户 A 打开用户端首页，进入活动详情并发起拼团。
   - 确认订单页展示“西北师大社区团购 / 社区拼团确认订单”。
   - 配送方式优先使用商家配送/快递地址。
   - 提交后生成 `trade_order.type = 5`，并写入 `group_buy_activity_id`、`group_buy_head_id`。

3. 用户 A 完成付款。
   - 支付结果页展示社区拼团成功提示。
   - “我的拼团”出现该团单，成员状态为已支付。

4. 用户 B 参与同一团单并付款。
   - 达到成团人数后，团单状态变为“已成团”。
   - 管理后台团单详情显示两个成员及对应订单。

5. 商家在管理后台交易订单中筛选“社区拼团订单”并发货。
   - 已成团订单允许发货。
   - 未成团订单必须被后端拒绝发货。

6. 用户端确认收货。
   - 订单状态进入已完成。
   - 订单详情仍可跳转拼团进度和活动详情。

## 4. 失败团退款

1. 创建一个短成团时限活动，发起拼团但不满足成团人数。
2. 触发 `GroupBuyHeadExpireJob` 或等待任务扫描。
3. 期望结果：
   - 团单状态变为“已失败”，失败原因显示“成团超时”。
   - 已支付成员调用交易订单整单退款，订单关闭原因使用拼团关闭。
   - 成员状态变为“已退款”。
   - 用户端“我的拼团”和订单详情能看到结束/关闭状态。
